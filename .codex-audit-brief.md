## Platform
CIX Sky1 (CP8180) arm64, Minisforum MS-R1 (`cixmini`, ~192.168.207.66). UEFI,
firmware often cannot persist NVRAM (rEFInd at EFI/BOOT/BOOTAA64.EFI fallback).
Console video = firmware efifb/simplefb; a KMS driver that seizes the panel
black-screens the box. Two kernels shipped: LTS `6.18.26-cix-sky1-lts`
(assets/kernel/stable) and EDGE `7.0.12-cix-sky1-next` (assets/kernel/edge).
rEFInd writes 4 menuentries (70-bootloader.sh): stable (LTS), edge (NEXT,
default), rescue (rescue.target on main root, LTS + safe blacklist), and a
dedicated on-disk RESCUE PARTITION (separate rootfs, populated by
72-rescue-partition.sh).

## SYMPTOM MATRIX after flashing the latest build (r130.6) and installing:
1. EDGE entry (DEFAULT, main system, 7.0.12, full display stack, NO blacklist):
   system DOES NOT BOOT — no video at all (not even the splash now), and ethernet
   never comes up. (In the prior r130.5 build it at least rendered the NCZ splash
   then hung; now nothing.)
2. RESCUE PARTITION (now LTS 6.18.26 + RESCUE_CMDLINE_SAFE which blacklists
   armchina_npu,panthor,mali,bifrost,cix_vpu,linlon_vpu,trilin_drm,trilin_dpsub,
   linlondp,linlondp_drv,cix_display): BOOTS, ethernet WORKS, but the SCREEN IS
   BLANK (no console video).
3. In the PREVIOUS build (r130.5) the RESCUE PARTITION ran the EDGE kernel with
   NEXT_CMDLINE_BASE (NO display blacklist) and HAD a working console.
4. The Plymouth watermark position fix (WatermarkVerticalAlignment=.08) works:
   logo now renders at the top. So plymouth itself renders on the edge path at
   least sometimes.

## My hypotheses (confirm/refute with file:line evidence):
H1 (rescue blank): adding RESCUE_CMDLINE_SAFE (which blacklists cix_display /
   linlondp / trilin_* / KMS) to the dedicated RESCUE PARTITION removed the only
   drivers that paint the Sky1 panel, so the console is blank. The standalone
   rescue PARTITION is meant to be a USABLE environment (operator logs in at the
   console), unlike the rescue.target pin (which recovers a broken main root). Fix:
   boot the rescue PARTITION with the NORMAL LTS cmdline (LTS_CMDLINE_BASE), NOT
   RESCUE_CMDLINE_SAFE — keep the LTS kernel swap (quiet/reliable) but KEEP the
   display drivers so there is a console. Confirm by reading 70-bootloader.sh
   RESCUEPART_OPTIONS and the cmdline definitions.
H2 (main edge no-boot): the EDGE 7.0.12 display/KMS stack wedges this MS-R1 panel
   → black screen AND the boot does not complete (so ethernet never initializes).
   This is the classic "edge boot misbehaves" the rescue exists for. If so, the
   pragmatic fix is to make the DEFAULT boot the LTS 6.18 kernel (which AGENTS.md
   calls "all drivers working") instead of edge, OR ship the edge entry with a
   display-safe cmdline. Audit: what differs between LTS_CMDLINE_BASE and
   NEXT_CMDLINE_BASE? Is there a known display/KMS workaround in one and not the
   other? Should default_selection be "stable" not "edge" on MS-R1?
