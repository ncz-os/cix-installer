# Stability Sweep 5 - Agent Stack and Container Path - 2026-05-08

## 1. Executive summary

- Verdict: high-severity first-boot agent activation gaps were present in `30-agents.sh` and the quadlet assets.
- Patched in-place: 3 HIGH fixes in `post-install/30-agents.sh` and `assets/agent-stack/*.container`; no commit made.
- Counts: HIGH 3, MEDIUM 7, LOW 5.
- Current post-patch behavior: zeroclaw, openclaw, and hermes are active rootful system quadlets on first boot, ordered behind a serialized image load/pull service.
- Remaining risk: the later `46-ncz-cli.sh` still overwrites the richer `ncz agent ...` CLI generated by `30-agents.sh`, and diagnostics still do not dump agent/container state.

## 2. HIGH findings

### H1 - Agent quadlets were not active on first boot, and netinstall could miss the asset path

File: `post-install/30-agents.sh:30`, `post-install/30-agents.sh:33`, `post-install/30-agents.sh:40`, `post-install/30-agents.sh:213`

Root cause: the hook had been converted to an opt-in model: it staged `.container`
files only as templates under `/usr/share/ncz/quadlets/`, deleted any active
quadlets under `/etc/containers/systemd/`, and expected `sudo ncz agent install`
to copy selected quadlets later. It also read templates from
`/cdrom/cixmini/assets/agent-stack`, but netinstall mode intentionally does not
bind-mount `/cdrom` into the target chroot.

Why HIGH: the requested deployment contract is "installed system boots and
agents come up." Pre-patch, a clean install could boot with no active
zeroclaw/openclaw/hermes units, and netinstall could have no staged templates
for the later installer path either.

Concrete diff applied:

- `30-agents.sh` now prefers `/usr/local/lib/cix-installer/assets/agent-stack`
  and only falls back to `/cdrom/cixmini/assets/agent-stack`.
- It no longer removes active zeroclaw/openclaw/hermes quadlets.
- It stages the three `.container` files and `hermes-isolated.network` into
  `/etc/containers/systemd/` on install.
- It still keeps copies under `/usr/share/ncz/quadlets/` for operator repair and
  reinstallation workflows.

### H2 - Image load/pull ordering was missing, so boot-time quadlets could race or fail independently

File: `post-install/30-agents.sh:106`, `post-install/30-agents.sh:116`, `post-install/30-agents.sh:124`, `post-install/30-agents.sh:185`, `assets/agent-stack/zeroclaw.container:4`, `assets/agent-stack/openclaw.container:4`, `assets/agent-stack/hermes.container:19`

Root cause: the hook explicitly removed references to
`nclawzero-load-agent-images.service` and disabled the service. The asset
quadlets only had `Wants=... nclawzero-load-agent-images.service`, not
`Requires=`, and there was no installed loader unit/script. With three services
starting from `multi-user.target`, Podman could attempt independent pulls, fail
slowly on cold-RTC TLS/network issues, or start before full-mode tarballs were
loaded.

Why HIGH: this is the race that turns a valid OS install into a non-deployment:
systemd reaches the boot target, but the agent stack is either absent or
degraded without a single authoritative image-load point.

Concrete diff applied:

- `30-agents.sh` now writes `/usr/share/ncz/agent-images.manifest`.
- It installs `/usr/local/sbin/nclawzero-load-agent-images`, which is
  idempotent: `podman image exists` skips already-present images, OCI tarballs
  under `/var/lib/nclawzero/agent-images/` are loaded when present, and missing
  tarballs fall back to registry pull.
- It installs `/etc/systemd/system/nclawzero-load-agent-images.service` as
  `Type=oneshot`, `RemainAfterExit=yes`, `ConditionPathExists=/var/lib/nclawzero/agent-images`,
  `TimeoutStartSec=3600`, and `Before=zeroclaw.service openclaw.service hermes.service`.
- The three quadlets now use `Requires=nclawzero-load-agent-images.service`,
  `After=network-online.target nclawzero-load-agent-images.service`, and
  `Pull=never`, so image acquisition is serialized through the loader.

Failure behavior after the patch: if registry pull fails on a netinstall first
boot, `nclawzero-load-agent-images.service` fails, the agent units fail, and the
system continues booting to a degraded but reachable target. Retrying after
network/time recovery is a normal `systemctl reset-failed ...; systemctl start
nclawzero-load-agent-images.service zeroclaw.service openclaw.service
hermes.service` operation.

### H3 - Zeroclaw referenced the wrong image source for this deployment

File: `assets/agent-stack/zeroclaw.container:9`, `post-install/30-agents.sh:118`, `post-install/30-agents.sh:236`

Root cause: the zeroclaw quadlet and generated `ncz` helper used
`ghcr.io/zeroclaw-labs/zeroclaw:latest`, while this sweep brief and the
existing asset comment pointed at the NCZ demo image,
`ghcr.io/perlowja/nclawzero-demo`.

Why HIGH: the loader service and quadlet must agree on the exact image ref.
Pulling the wrong zeroclaw image can produce a container that starts but does
not expose the expected NCZ demo surface on port 42617.

Concrete diff applied:

- `assets/agent-stack/zeroclaw.container` now uses
  `ghcr.io/perlowja/nclawzero-demo:latest`.
- The generated manifest in `30-agents.sh` uses the same ref.
- The generated `ncz` helper image map and desktop quick-reference text were
  updated to the same ref.

## 3. MEDIUM findings

### M1 - `46-ncz-cli.sh` overwrites the richer agent CLI generated by `30-agents.sh`

File: `post-install/30-agents.sh:219`, `post-install/30-agents.sh:236`, `post-install/46-ncz-cli.sh:38`, `post-install/46-ncz-cli.sh:58`

Root cause: hook order runs `30-agents.sh` before `46-ncz-cli.sh`. The first
hook writes a larger `/usr/local/bin/ncz` with `agent install/list/status/logs`
commands. The later hook rewrites `/usr/local/bin/ncz` with a different CLI
surface that has `desktop`, `models`, `install mnemos`, `status`, `version`, and
`help`, but no `agent` subcommand.

Impact: after this patch the core agents no longer depend on `ncz agent install`
to come up, but the operator-facing commands and help text from `30-agents.sh`
are still lost. This directly weakens the debug surface and makes docs that say
`ncz agent list` incorrect on the installed target.

Concrete diff recommendation: merge the `agent` subcommands into the
authoritative `46-ncz-cli.sh` version, or have `30-agents.sh` install a
separate helper such as `/usr/local/bin/ncz-agent` and make `46-ncz-cli.sh`
dispatch `ncz agent ...` to it. I did not patch `46-ncz-cli.sh` in this sweep
because it was outside the named file scope.

### M2 - Magnetar does not skip desktop hooks; it only masks the desktop later

File: `post-install/48-magnetar-variant.sh:18`, `post-install/48-magnetar-variant.sh:25`, `post-install/48-magnetar-variant.sh:42`, `post-install/48-magnetar-variant.sh:49`

Root cause: variant detection itself is robust: it reads
`/usr/local/lib/cix-installer/BUILD_VARIANT`, accepts `desktop|reinhardt` and
`server|magnetar|headless`, and does not guess from hostname. The gap is that
this hook runs after earlier optional hooks have already installed desktop,
browser, branding, and launcher material. For Magnetar it sets
`multi-user.target`, masks display managers, and enables SSH/console affordances,
but it does not prevent desktop hooks from running.

Impact: Magnetar boots headless, but it is not "agents-only, no desktop" in the
package/image sense. This is a SKU hygiene and attack-surface issue, not a
first-boot agent failure.

Concrete diff recommendation: add a variant-aware skip layer in the
orchestrator or individual desktop hooks so Magnetar omits desktop/browser
payloads before they are installed. Do not infer variant from hostname.

### M3 - `99-diagnostics.sh` does not capture the requested agent debug state

File: `post-install/99-diagnostics.sh:89`, `post-install/99-diagnostics.sh:93`, `post-install/99-diagnostics.sh:94`

Root cause: final diagnostics capture `systemctl --failed` and enable/active
state for SSH/network services only. They do not run `podman ps`, `podman
images`, `systemctl status zeroclaw openclaw hermes
nclawzero-load-agent-images`, or any port-binding check.

Impact: if the loader or any agent unit fails after first boot, the install-time
diagnostic bundle lacks the exact evidence the operator needs.

Concrete diff recommendation: add `agent-state.log` with:

```sh
podman ps -a
podman images
systemctl status --no-pager nclawzero-load-agent-images zeroclaw openclaw hermes
ss -ltnp | grep -E ':(42617|18789|8642)\b'
journalctl -u nclawzero-load-agent-images -u zeroclaw -u openclaw -u hermes --no-pager -n 200
```

### M4 - Embedkit is not actually offline-complete in this checkout

File: `post-install/47-embedkit.sh:40`, `post-install/47-embedkit.sh:52`, `post-install/47-embedkit.sh:58`, `post-install/47-embedkit.sh:77`, `post-install/47-embedkit.sh:88`

Root cause: the hook prefers a bundled `mnemos_embedkit-*.whl` or source repo,
but `assets/embedkit/` is empty in this checkout. It then installs
`llama-cpp-python` from an external wheel index. The `.cix` model is referenced
and copied if present, but only the `.gguf` model is present locally.

Impact: on an offline full install, embedkit can silently degrade to a venv with
missing package pieces. On netinstall, success depends on Python package network
availability and Python 3.13 wheel compatibility. The hook is idempotent in the
file-copy sense, but because it runs without `set -e`, failed venv/pip work can
still leave exit 0.

Concrete diff recommendation: ship the embedkit wheelhouse, add `python3-venv`
as an explicit dependency, avoid network installs during the hook, and fail if
the expected package import is missing unless a documented "defer embedkit"
mode is set.

### M5 - LLM stack hook installs substrate tools, not llama.cpp bringup

File: `post-install/47-llm-stack.sh:31`, `post-install/47-llm-stack.sh:42`, `post-install/47-llm-stack.sh:75`, `post-install/47-llm-stack.sh:77`

Root cause: `47-llm-stack.sh` installs Vulkan/SPIR-V tooling and diagnostic
utilities. It explicitly defers llama.cpp binaries, model loadout, and NPU
embedder server wiring to a later `ncz install mnemos` path.

Impact: the hook name and sweep brief imply local LLM bringup, but the current
implementation is only a substrate bake. Mesa/panvk pinning is owned by
`15-mesa-sky1-pin.sh`, not by this hook. Mali firmware is owned by
`12-sky1-firmware.sh`.

Concrete diff recommendation: either rename/status-message the hook as
substrate-only, or add the actual llama.cpp Vulkan binary and a non-fatal
first-boot smoke unit that confirms `vulkaninfo --summary` and a tiny model run.

### M6 - Fleet env samples disagree on provider policy and absent-key behavior

File: `post-install/30-agents.sh:73`, `post-install/30-agents.sh:80`, `post-install/30-agents.sh:87`, `assets/agent-stack/agent-env.sample:8`, `assets/agent-stack/agent-env.sample:19`

Root cause: `30-agents.sh` writes an inline sample with eight keys including
`ANTHROPIC_API_KEY` and `MISTRAL_API_KEY`. The asset sample says Anthropic is
forbidden, lists seven providers, and adds `NVIDIA_API_KEY`, `XAI_API_KEY`, and
`ZEROCLAW_DEFAULT_MODEL`. The live file is created mode `0640`, group
`nclawzero`, which matches the sensitivity requirement, but its content comes
from the inline sample rather than the asset sample.

Impact: file mode is correct, but provider policy is inconsistent and empty
keys are accepted silently. Agents may start yet fail at first provider call.

Concrete diff recommendation: make `assets/agent-stack/agent-env.sample` the
single source of truth, copy it from `30-agents.sh`, and add an operator-visible
status check that reports "no fleet keys configured" without printing values.

### M7 - Hermes exposure comments and active behavior diverge

File: `assets/agent-stack/hermes.container:40`, `post-install/30-agents.sh:57`, `post-install/30-agents.sh:59`

Root cause: the checked-in `hermes.container` documents loopback-only exposure
and has `PublishPort=127.0.0.1:8642:8642`, but `30-agents.sh` rewrites the
template to `PublishPort=8642:8642` for LAN exposure before staging active
quadlets.

Impact: this is not a port collision, but it is a security posture/documentation
mismatch. Operators reading the asset file will expect Hermes to be local-only,
while the installed target exposes it on all host interfaces.

Concrete diff recommendation: either update the asset to the LAN posture and
document the risk, or remove the sed rewrite and keep Hermes loopback-only by
default.

## 4. LOW findings and recommendations

### L1 - Quadlet shim no-op is acceptable only for Podman 4.4+

File: `post-install/32-quadlet-shim.sh:9`, `post-install/32-quadlet-shim.sh:12`, `post-install/32-quadlet-shim.sh:16`

The hook detects pre-4.4 Podman and logs that it would need a shim, but still
skips. This is acceptable if the base image contract is Podman 5.x. Add a hard
preflight in `30-agents.sh` if older Podman ever re-enters the matrix.

### L2 - `30-agents.sh` remains mostly best-effort under `set +e`

File: `post-install/30-agents.sh:23`, `post-install/30-agents.sh:26`, `post-install/30-agents.sh:41`

The hook is optional in `run-all.sh`, so best-effort behavior is expected. For a
release gate, though, missing Podman or missing agent assets should probably
write a machine-readable failure marker under `/var/log/cix-install/` even if
the installer continues.

### L3 - Agent image loader is generated inline and lacks standalone linting

File: `post-install/30-agents.sh:124`, `post-install/30-agents.sh:185`

The new loader script and systemd unit are generated from heredocs. This
matches local style in `30-agents.sh`, but moving them into
`assets/agent-stack/` would make shellcheck, unit review, and hash inventory
easier.

### L4 - Full-mode OCI tarball inventory is not present in the repo checkout

File: `post-install/30-agents.sh:110`, `post-install/30-agents.sh:116`

The patch supports future tarballs under
`/usr/local/lib/cix-installer/assets/agent-images`, copying them into
`/var/lib/nclawzero/agent-images`. This checkout has no such tarballs, so the
audit cannot verify full-mode offline image load from repo contents alone.
Add an ISO payload manifest with sha256 sums for each `.oci.tar`.

### L5 - Portainer is still separate from the three-agent contract

File: `post-install/30-agents.sh:239`, `post-install/30-agents.sh:287`

The three core agents are now active. Portainer remains a direct `podman run`
path behind `ncz agent install portainer`. That is fine for this sweep because
the requested invariant names zeroclaw, openclaw, and hermes only.

### Verify H2 - image loading is serialized and idempotent

```sh
sudo test -f /usr/share/ncz/agent-images.manifest
sudo test -x /usr/local/sbin/nclawzero-load-agent-images
systemctl cat nclawzero-load-agent-images.service
systemctl show -p Type -p RemainAfterExit -p TimeoutStartUSec -p ConditionResult nclawzero-load-agent-images.service

systemctl show -p Requires -p After zeroclaw.service
systemctl show -p Requires -p After openclaw.service
systemctl show -p Requires -p After hermes.service

sudo systemctl restart nclawzero-load-agent-images.service
sudo systemctl restart nclawzero-load-agent-images.service
journalctl -u nclawzero-load-agent-images.service --no-pager -n 120
```

Expected:

- The unit is `Type=oneshot`, has `RemainAfterExit=yes`, and uses a 3600 second
  timeout.
- Each agent service requires and starts after the loader service.
- The second loader restart prints "already present" for loaded/pulled images
  or otherwise does no destructive work.

For full/offline mode with tarballs:

```sh
sudo ls -lh /var/lib/nclawzero/agent-images
sudo journalctl -u nclawzero-load-agent-images.service --no-pager | grep -E 'loading .* from /var/lib/nclawzero/agent-images'
```

For netinstall mode with no tarballs:

```sh
sudo journalctl -u nclawzero-load-agent-images.service --no-pager | grep -E 'no OCI tarball.*pulling'
```

To verify clean registry-failure behavior, boot once with network unavailable:

```sh
systemctl is-system-running || true
systemctl --failed
systemctl status nclawzero-load-agent-images zeroclaw openclaw hermes --no-pager
```

Expected: boot reaches a usable target, the loader/agent units are failed, and
SSH/console still work. After restoring network/time:

```sh
sudo systemctl reset-failed nclawzero-load-agent-images zeroclaw openclaw hermes
sudo systemctl start nclawzero-load-agent-images.service
sudo systemctl start zeroclaw.service openclaw.service hermes.service
```
