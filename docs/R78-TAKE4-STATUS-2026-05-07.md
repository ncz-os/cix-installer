## What broke on r78-take3

Symptom on real .66:
- `installer@<host>` with password `Gumbo@Kona1b` worked (network-
  console came up).
- `root@<host>` with the fleet pubkey did NOT work.
- `magnetar@<host>` did NOT exist on the installed system.

Codex audit (`docs/R78-QEMU-DEBUG-2026-05-07.md`) identified:

1. **Root SSH gotcha.** `preseed/early_command` patched
   `/etc/ssh/sshd_config` for `PermitRootLogin yes` only if the file
   existed at preseed-load time. But `network-console` (which installs
   sshd via udebs) loads later. So the sed was a no-op and root SSH
   was never enabled.

2. **Magnetar account never created.** `09-diag-account.sh` (which
   creates the magnetar diag user) lived in `post-install/run-all.sh`
   Phase 2 (optional). If `10-our-kernel.sh` aborted in Phase 1
   (required), `run-all.sh` exited before reaching Phase 2. Magnetar
   was therefore conditional on the kernel install succeeding.

3. **QEMU wrapper mismatches** (separate concern, see
   `R78-QEMU-DEBUG-2026-05-07.md` section "QEMU wrapper gotchas").

### Part 1 — `preseed/preseed-ubuntu.cfg`

Rewrote `d-i preseed/early_command`. Now writes
`/root/.ssh/authorized_keys` with the fleet pubkeys (jperlow @ work-
laptop and argos-backup), then writes
`/etc/early_command_ran.txt` as a sentinel, then spawns
`/tmp/sshd-watcher.sh` via nohup. The watcher polls for both
`/etc/ssh/sshd_config` AND a running sshd process up to 600 seconds.
When both are present, it patches `PermitRootLogin yes` +
`PubkeyAuthentication yes`, kills -HUP sshd, and tries
`/etc/init.d/ssh restart` + `reload` for good measure.

Watcher logs to `/var/log/early_command.log` so we can audit what
happened post-install.

### Part 2 — `post-install/run-all.sh`

Added a Phase 0 that runs hooks matching `0[0-9]-*.sh` BEFORE Phase 1
(kernel install). `set +e` so it's non-blocking. This guarantees
`09-diag-account.sh` runs before any required hook can abort.

Phase 2's `OPT_HOOKS` regex was updated to exclude `0[0-9]` so Phase 0
hooks don't double-run.

## Why QEMU verification didn't run

Attempted on ARGOS following the Codex-recommended invocation in
`R78-QEMU-DEBUG-2026-05-07.md` section "Recommended ARGOS QEMU
command". Killed after 40s of no useful output.

Three structural mismatches:

1. The d-i kernel was swapped to `linux-cix-sky1` NEXT (7.0.3) at
   bake time. That kernel was compiled for Cix Sky1 hardware. Its
   initramfs likely lacks `virtio_blk` / `virtio_net` modules that
   QEMU virt needs.
2. GRUB cmdline hard-codes `console=ttyAMA2,115200`. QEMU virt's only
   PL011 UART is at ttyAMA0. With `-serial null -serial null -serial
   mon:stdio`, the kernel's serial output landed nowhere we could
   capture.
3. `preseed/preseed-ubuntu.cfg` hard-codes `/dev/nvme0n1` for
   partman. We exposed an NVMe disk, so partman would have found it,
   but the upstream issue was already going to bite first.

A QEMU-suitable bake would need: (a) a different kernel cmdline
(`console=ttyAMA0,115200 console=tty0`), (b) a different installer
kernel (the bookworm-12 generic-arm64 one or a Cix kernel with
`CONFIG_VIRTIO_BLK=y` baked), (c) optionally a partman-agnostic
preseed. None of those are in scope for an install bug-fix bake.

Real-hardware verification on .66 is the canonical path.

## Why a second round was needed (post-take4 self-audit)

After take4 was baked I extracted the ISO and ran a deeper probe of
the d-i busybox initrd and the preseed semantics:

```sh
cd /tmp/iso-extract
xorriso -osirrox on -indev TAKE4.iso \
    -extract /install.a64/initrd.gz initrd.gz
mkdir initrd-mnt && cd initrd-mnt
zcat ../initrd.gz | cpio -id

./bin/busybox --list | grep -E '^(nohup|setsid|disown)$'
# (empty — none of those are applets in busybox 1.35)
find . -name 'nohup*' -o -name 'setsid*'
# (empty — no binaries either)
```

That made it obvious the watcher's `nohup ... &` was a no-op. From
there I re-read the early_command block and realized the heredoc
embedding wouldn't survive d-i's preseed-parser line-continuation
collapse — heredoc body lines don't end in backslash.

Extracting `/cixmini/preseed.cfg` from the take4 ISO showed the
heredoc was preserved in the file as written, but that's the source
representation — not what d-i's parser produces after collapsing
continuations into a single shell command.

Both issues are addressed in commit `1c6153c`. take5 ISO is rebaking
now.
